# Zero-Budget Engineering Playbook
## Когда денег нет, но есть техническая сила

Свод выходов из ситуаций «упёрлись в free-tier лимит / нет ресурсов». Рождён из боли UzMRC (эмбеддер-лимиты), но применим везде. Порядок — от самого дешёвого по усилиям к самому мощному.

---

## Tier 0 — Вообще не делать вызов (убрать спрос)
Самый дешёвый «безлимит» — не обращаться к лимитируемому ресурсу.

- **Кэш результатов по хэшу контента** ⭐ — `sha256(provider+model+text) → vector/ответ`, в Postgres/файл. Эмбеддинг/LLM считается ОДИН раз навсегда; повторная индексация — мгновенна и бесплатна. **Это наш #1 невзятый фикс** (мы сегодня жгли квоту 4 реиндексами одного и того же корпуса).
- **Precompute & ship** — посчитать эмбеддинги корпуса один раз оффлайн, сохранить векторы как артефакт (в репо/БД/файл), грузить готовые. Корпус фиксирован → платить за эмбеддинг ровно один раз.
- **Батчинг** — N вызовов → 1 (мы сделали для судьи: 45→6, и для эмбеддинга запросов).
- **Идемпотентный + резюмируемый пайплайн** — чекпоинты, чтобы прерывание ≠ старт с нуля. Наша боль: воркер на `BackgroundTasks` умирал при рестарте → реиндекс с нуля. Кэш (Tier 0) делает резюм бесплатным.

## Tier 1 — Self-host на бесплатном железе (меняем тех-скилл на $0) ⭐
- **Oracle Cloud Always Free** — **ARM VM 12–24GB RAM, навсегда бесплатно** (с 15 июня 2026 урезали до 2 ядер/12GB, но это всё равно >>0.89GB ноута). Хостим self-hosted **bge-m3 через TEI/Infinity** + хоть весь стек. **Решает проблему RAM**, которая блокировала локальный эмбеддер. GPU нет, но CPU-эмбеддинги тянет. → нулевые лимиты, on-prem-подобно. *(до прихода сервера клиента — это и есть «сервер»)*
- **Бесплатный GPU для разовых тяжёлых задач**: **Kaggle — 30 GPU-часов/неделю** (T4/P100 16GB, надёжно), Colab 15–30ч/нед. Прогнать **bulk-эмбеддинг всего корпуса bge-m3 на free-GPU → выгрузить векторы → импорт в Qdrant**. API-лимитов нет вообще. Идеально для фиксированного корпуса.
- **Локально (ноут)**: `fastembed` (ONNX CPU, e5-small ~470MB), Ollama (`nomic-embed`/квант LLM) — когда RAM позволяет. Квантизация Q4 чтобы влезть.

## Tier 2 — Размазать по free-tier'ам (горизонтальное масштабирование бесплатного)
- **Свой провайдер на каждую роль** — эмбеддинги / LLM-chat / rerank / vision у разных бесплатных провайдеров (их лимиты независимы). Мы уже: Voyage (embed) + Cerebras/Gemini (LLM).
- **Кросс-провайдер фолбэк** — primary падает по квоте → secondary подхватывает (мы построили Cerebras↔Gemini для судьи).
- **Ротация ключей** — несколько бесплатных аккаунтов/ключей, round-robin → умножаем эффективный RPM/RPD. (Классический приём для Gemini.)
- **Выбирать провайдера под ФОРМУ лимита**: токен-бюджет/мес vs RPD vs RPM. Под наш bulk-ингест важен «нет дневного cap».

## Tier 3 — Уважать лимит технично (resilience)
- **Клиентский rate-limiter** (RPM+TPM, token-batching) — мы написали в `voyage.py`/`embeddings.py`. Делает любой free-эмбеддер пригодным для полного ингеста.
- **Терпеливый retry с Retry-After** + экспон. backoff.
- **Off-peak / ночью** — тяжёлый ингест в фоне, когда не мешает.

## Tier 4 — Уменьшить след
- **Меньшая размерность** (matryoshka: gemini/Jina умеют 1024/768 вместо 3072) — меньше хранилища/токенов.
- **Крупнее чанки / меньше чанков** — меньше эмбеддинг-вызовов (баланс с гранулярностью retrieval).

## Что применить НАМ (приоритет)
1. **Embedding cache** (Tier 0) — немедленно. Останавливает пере-жигание квот при реиндексе; делает итерации мгновенными. Самый высокий ROI.
2. **Jina v4 как дев-эмбеддер** (Tier 2) — 1M ток/мес + 2000 RPM, нет дневного cap → быстрый полный ингест без боли Voyage (3 RPM) и Gemini (1000/день). Нужен free-ключ jina.ai.
3. **Oracle Cloud Always Free VM** (Tier 1) — поднять self-hosted bge-m3 (и/или весь стек). Прод-grade, без лимитов, до прихода сервера клиента. Решает RAM навсегда.
4. **Kaggle free-GPU** (Tier 1) — если хотим bge-m3 качество без хостинга: разово эмбеднуть корпус на GPU → выгрузить векторы.

**Вывод:** «нет денег, есть скилл» = три рычага — **(а) не вызывать (кэш/precompute), (б) self-host на бесплатном железе (Oracle/Kaggle), (в) размазать по free-tier'ам (мульти-провайдер/ротация/фолбэк) + технично уважать лимит (rate-limiter).** Мы пока использовали только (в)+часть Tier3. Самый недоиспользованный и мощный — **кэш (Tier 0) + Oracle free VM (Tier 1)**.
