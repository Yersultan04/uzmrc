# UzMRC — Eval-харность

Измеримое качество обоих модулей MVP. Гоняется против живого стека (`docker compose ... up`).

## Запуск

```bash
cd eval
python run_eval.py                 # обе сюиты (compare усредняется по 3 прогонам)
python run_eval.py --only retrieval --delay 5
python run_eval.py --only compare --runs 5
```

Конфиг через env (дефолты под локальный стек):
`UZMRC_API` (http://127.0.0.1:8088), `UZMRC_RAG_ID`, `UZMRC_EMAIL`, `UZMRC_PASS`.

## Метрики

**Модуль 1 — Retrieval** (`retrieval_gold.jsonl`): размеченные запросы ru/uz → релевантные доки.
- `recall@1/3/5` — попал ли релевантный док в топ-k.
- `MRR` — средний обратный ранг первого релевантного.
- `--delay` между запросами уважает RPM бесплатного эмбеддера (Voyage).

**Модуль 2 — Compare** (`compare_gold.jsonl`): 10 положений демо-приказа с известными по построению типами.
- `accuracy` — доля положений, чей предсказанный тип попал в допустимое множество (`acceptable`).
- Несколько прогонов (`--runs`) усредняются, т.к. LLM-судья стохастичен.
- Матрица ошибок gold→predicted показывает, где модель путает типы.
- `acceptable` содержит мягкие границы (дубль↔дополнение, конфликт↔пробел), чтобы не штрафовать защитимые ответы.

## Файлы
- `retrieval_gold.jsonl` — 16 запросов (ru+uz) → релевантные имена доков.
- `compare_gold.jsonl` — 10 положений → `primary` + `acceptable` метки.
- `run_eval.py` — раннер (stdlib-only).
